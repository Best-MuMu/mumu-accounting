# 暮暮记账本 — 产品文档

---

## 一、项目概述

| 项目 | 内容 |
|------|------|
| **APP 名称** | 暮暮记账本 |
| **标语** | 每一笔，都算数 |
| **定位** | 轻量、离线优先的个人桌面记账工具 |
| **目标用户** | 学生、上班族、自由职业者等需要记录日常个人支出的用户 |
| **运行平台** | Windows 10+ 和 macOS 11+ |
| **货币单位** | 人民币（¥） |
| **网络依赖** | 无 — 完全离线可用，无需账号、无需登录 |

---

## 二、核心功能（分阶段）

### 第一阶段：MVP（最小可用版本）
1. **记录支出** — 输入金额、选择分类、选择日期、可选备注
2. **支出历史** — 分页列表，按日期倒序排列，支持按日期范围和分类筛选
3. **分类管理** — 可浏览完整分类树，用户可添加自定义二级分类（一级分类固定）
4. **月度统计** — 本月总支出、日均支出、支出最多分类、近30天每日支出柱状图
5. **数据本地存储** — 所有数据存于本地，无需网络

### 第二阶段：增强功能（MVP 之后）
6. 高级统计 — 分类饼图、环比对比、年度趋势折线图
7. 数据导出 — 导出为 CSV 或 Excel 文件
8. 数据导入 — 从 CSV 文件导入
9. 预算提醒 — 按一级分类设置月度预算，接近上限时提醒
10. 周期性支出 — 标记周期性支出（如每月房租），按时自动生成记录
11. 全文搜索 — 搜索备注和分类
12. 深色模式 — 浅色/深色主题切换

---

## 三、UI/UX 设计原则

1. **速度优先** — 记录一笔支出不超过 10 秒。默认日期为"今天"，默认选中常用分类，打开即聚焦金额输入框
2. **清晰简洁** — 每屏一个主要操作，留白充足，排版清晰
3. **人民币原生** — ¥ 符号始终可见，金额保留两位小数
4. **键盘优先** — 核心操作均可通过键盘完成（Tab 切换、Enter 确认、Esc 取消）
5. **离线优先** — 任何功能都不依赖网络请求
6. **分类为核心** — 分类选择器是记录流程的核心交互元素

## 五、数据模型

### 支出记录

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | 字符串 (UUID) | 是 | 唯一标识，创建时自动生成 |
| `amount` | 整数 | 是 | 金额，以"分"为单位存储（如 ¥12.50 存为 `1250`），避免浮点数计算误差 |
| `categoryL1` | 字符串 | 是 | 一级分类名称（如 "餐饮"） |
| `categoryL2` | 字符串 | 是 | 二级分类名称（如 "午餐"） |
| `date` | 字符串 | 是 | 支出日期，格式 `YYYY-MM-DD`（如 "2026-07-21"） |
| `note` | 字符串 | 否 | 备注，最多 200 字 |
| `createdAt` | 字符串 | 是 | 记录创建时间，ISO 8601 格式 |
| `updatedAt` | 字符串 | 是 | 记录最后修改时间，ISO 8601 格式 |

### 分类配置

| 字段 | 类型 | 说明 |
|------|------|------|
| `l1` | 字符串 | 一级分类名称（10 个固定值） |
| `l2` | 字符串数组 | 该一级分类下的二级分类名称列表 |

## 六、技术方案 ✅ 已确定

**最终选择：方案 C — 本地 Web 服务 + 浏览器**（用户于 2026-07-22 最终确认）

> 初始选择 Tauri → 因 Rust 工具链缺失切换至 Electron → 因 Electron 二进制下载源不可用（GitHub 不可达、镜像源文件异常），最终采用本地 Web 服务方案。

| 技术 | 用途 |
|------|------|
| **Node.js + Express** | 本地 Web 服务器（提供 API 接口） |
| **React** | 前端 UI 框架（构建用户界面） |
| **TypeScript** | 编程语言（类型安全检查） |
| **SQLite (sql.js)** | 本地数据库（纯 JS 实现，无需编译） |
| **Vite** | 前端构建工具（开发热更新 + 生产打包） |

**使用方式**：运行 `npm start`，自动打开浏览器访问 `http://localhost:3000`

| 技术 | 用途 |
|------|------|
| **Electron** | 桌面应用框架（将 Web 界面封装为原生桌面应用） |
| **React** | 前端 UI 框架（构建用户界面） |
| **TypeScript** | 编程语言（在 JavaScript 基础上增加类型安全检查） |
| **SQLite** (better-sqlite3) | 本地数据库（数据存储在用户电脑上的单个文件） |
| **Node.js** | Electron 后端运行时（处理数据库操作和文件读写） |

> 注：最初选择了 Tauri，但因 Windows 环境下 Rust GNU 工具链缺少 dlltool 组件，且 MinGW-w64 下载失败，用户决定切换至 Electron（Node.js 环境已就绪，可立即开始开发）。

### 方案对比回顾

### 方案 A：Electron + React + TypeScript + SQLite

> **通俗解释**：把一个精美的网页打包进一个迷你 Chrome 浏览器，让它像原生桌面应用一样运行。VS Code、Slack、Notion 都是这么做的。

| 维度 | 评价 |
|------|------|
| 📦 安装包大小 | 150–200 MB（因为自带一整个浏览器） |
| 💾 内存占用 | 150–300 MB |
| ⚡ 启动速度 | 中等（1–2 秒） |
| 🎨 界面精美度 | ⭐⭐⭐⭐⭐ 可以做非常精美的界面 |
| 📚 生态成熟度 | ⭐⭐⭐⭐⭐ 最成熟，社区最大 |
| 🔒 数据安全 | ⭐⭐⭐⭐⭐ SQLite 文件存储，稳定可靠 |
| 🛠 开发难度 | 中等 |

**优点**：生态最成熟，问题容易解决；UI 可以做到像素级精美；SQLite 集成非常成熟。
**缺点**：安装包大、内存占用高、启动稍慢。

---

### 方案 B：Tauri + React + TypeScript + SQLite ⭐ 推荐

> **通俗解释**：利用 Windows/Mac 系统自带的内置浏览器内核来渲染界面（不需要单独打包浏览器），后端用 Rust 语言处理数据和文件操作。安装包只有 Electron 的十分之一不到。

| 维度 | 评价 |
|------|------|
| 📦 安装包大小 | 5–15 MB（极小！） |
| 💾 内存占用 | 50–100 MB |
| ⚡ 启动速度 | 极快（秒开） |
| 🎨 界面精美度 | ⭐⭐⭐⭐⭐ 同样使用 Web 技术做界面 |
| 📚 生态成熟度 | ⭐⭐⭐⭐ 较成熟，社区快速增长中 |
| 🔒 数据安全 | ⭐⭐⭐⭐⭐ SQLite 文件存储，稳定可靠 |
| 🛠 开发难度 | 中等偏高（Rust 后端需学习，但本项目后端逻辑简单） |

**优点**：安装包极小、内存占用低、启动飞快、安全性设计优秀。
**缺点**：生态比 Electron 年轻一些；后端用 Rust 语言（但本项目的后端逻辑非常简单，影响不大）。

---

### 方案 C：Flutter Desktop

> **通俗解释**：Google 出品的跨平台框架，用自己的"画板"绘制界面（不依赖浏览器），一套代码编译出 Windows 和 Mac 应用。

| 维度 | 评价 |
|------|------|
| 📦 安装包大小 | 30–50 MB |
| 💾 内存占用 | 60–120 MB |
| ⚡ 启动速度 | 快 |
| 🎨 界面精美度 | ⭐⭐⭐⭐ 有自己的 Material Design 风格 |
| 📚 生态成熟度 | ⭐⭐⭐ 桌面端仍在成熟中 |
| 🔒 数据安全 | ⭐⭐⭐⭐ 良好 |
| 🛠 开发难度 | 中等（Dart 语言，桌面端文档较少） |

**优点**：界面在 Windows 和 Mac 上完全一致；内置丰富的组件。
**缺点**：桌面端生态还在成熟期，可能遇到插件不兼容；界面不像原生 Windows/Mac 应用；Dart 语言社区较小。

---

### 方案 D：PWA（网页应用）❌ 不推荐

> **通俗解释**：在浏览器中运行的网页，可以"安装"到桌面，但本质还是个网站。

| 维度 | 评价 |
|------|------|
| 📦 安装包大小 | 1–2 MB |
| 💾 内存占用 | 取决于浏览器 |
| ⚡ 启动速度 | 快 |
| 🎨 界面精美度 | ⭐⭐⭐ 受限于浏览器 |
| 📚 生态成熟度 | ⭐⭐⭐⭐⭐ |
| 🔒 数据安全 | ⚠️ 数据存浏览器中，清除浏览器数据可能丢失 |
| 🛠 开发难度 | 低 |

**优点**：无需安装，打开网址就能用。
**缺点**：⚠️ **数据存浏览器中，清除浏览器数据会导致记账数据丢失（不可接受！）**；没有真正的桌面体验；需要服务器托管。

---

### 我的推荐

**首选方案 B（Tauri）** — 安装包小（5–15 MB）、内存占用低（50–100 MB）、启动快、数据安全可靠。本项目的后端逻辑简单（就是增删改查），Rust 部分工作量很小。

**备选方案 A（Electron）** — 如果更看重生态成熟度和开发速度，愿意接受 150–200 MB 的安装包。

**不推荐方案 D（PWA）** — 记账数据有丢失风险，不可接受。

## 七、⚠️ 关键协作规则（整个项目期间必须遵守）

**用户是非技术背景，初次使用 Claude Code。**

对于项目开发过程中的**任何技术决策**，必须遵循以下规则：

1. **绝不替用户做技术假设** — 即使某个选择看起来"显然正确"，也必须列出方案让用户决定
2. **每个决策提供 2–3 个具体方案** — 用通俗语言解释每个方案是什么
3. **列出每个方案的优点和缺点** — 用表格或列表呈现
4. **给出推荐并说明理由** — 用户可以采纳或拒绝
5. **由用户最终拍板** — 用户不选择就不推进

### 需要用户决策的技术问题示例：
- 选择哪个技术栈
- 选择哪个 UI 组件库 / 图标库
- 金额存储用"分"（整数）还是"元"（小数）
- 颜色方案和视觉主题
- 字体选择
- 安装包格式（Windows：exe/msi，Mac：dmg/pkg）
- APP 图标设计方向

### 不需要用户决策的问题示例：
- 代码中变量怎么命名
- 代码格式化和缩进
- 项目文件夹怎么组织
- npm/cargo 包的版本号
- 测试代码的结构

### 决策模板：

>
> **选项 A：[名称]**

>
> **选项 B：[名称]**

>
